# 每月調價建議系統

從 BigQuery 資料倉儲直接讀全通路銷售，產出全品牌的調價指令表（漲/降/不動/出清），
以「毛利率 = 1/|彈性|」的利潤最大化原則小步逼近最佳價，每月最多調 ±7%。

## 資料來源（不用再手動匯出銷貨明細）

- **銷售/品牌/類別**：BigQuery `igogo-sales-dw.sales`（v_sales 全通路 + dim_item），
  金鑰 `C:\Users\Eric\.claude\secrets\bq_sa.json`。之後 DW 會有 agent 自動更新。
- **定價/毛利/彈性**：只取 **MOMO（富邦媒體）+ PChome（網路家庭）** 兩通路
  （config 的 `pricing_customers`）——因為 15%/10% 抽成假設與 NO_USER1 頁面價
  只適用這兩個。其他電商（官網、蝦皮、好市多線上）抽成不同，混入會失真。
- **月均銷量（庫存消耗速度）**：用**全通路**，庫存月數才準。
- **建議售價**：ECOUNT API 的 NO_USER1（見下）。
- **現有庫存**：優先讀 BigQuery `sales.v_inventory`（每日快照、含安全庫存/庫存金額，
  由 erp 儀表板的 sync_inventory.py 每日更新）；失敗才回退 ECOUNT 即時 API。
  庫存出清判斷已啟用。inventory_snapshot 有每日歷史，可看庫存趨勢。
- **成本（選用強化）**：`data/cost/` 若放庫存成本報表，成本用其移動平均（比 DW in_price 準）；
  沒放則用 dim_item.in_price。整套已不需手動匯出任何檔即可跑。

## 競品錨定（漲價天花板）

- `data/competitor/` 放 PCHome+MOMO 藍牙耳機競品快照（品牌/型號/價格/形態/平台）。
- 每個耳機 SKU 依**形態＋相近價位**（±window）對到同類競品，算市場中位/低標，
  排除我方品牌（JLab/EarFun/Cleer）避免自己錨自己。
- 漲價建議售價不得超過「同形態市場中位 × comp_ceiling_factor(1.05)」；
  已在市場價的品項改判「不動」，理由註明競品天花板。
- 儀表板新增「市場中位」「vs市場」欄與詳情面板定位。
- 資料源：優先讀 BigQuery `sales.competitor_prices` 的**最新月爬蟲快照**，
  無則回退 `data/competitor/` 的 xlsx。

## 競品爬蟲（每月自動累積）

- `crawl_competitors.py`：爬 MOMO(JSON-LD) + PChome(API) 各形態關鍵字，
  正規化品牌/形態，以 scan_date 累積寫入 BigQuery `sales.competitor_prices`。
- **排程**：Windows 工作排程器「igogo競品爬蟲月更」，每月 1 號 10:07，
  **StartWhenAvailable=True**（1號沒開機 → 下次開機自動補跑）。
  執行檔 `run_crawl.cmd`，log 在 `cache/crawl_log.txt`。
- 累積 3~6 個月後，即可用 SQL 把「你的價格/銷量(v_sales) + 競品價格」對齊，
  估**交叉彈性/權重 w**（每個商品該盯競品還是盯自身彈性）。目前是第 1 個月。

## 建議售價來源：ECOUNT API

- 「目前售價」＝ ECOUNT 品目主檔的自訂欄位 **NO_USER1**（官方定價），
  透過 API 自動抓取（金鑰從 `C:\Users\Eric\.claude\secrets\igogo.env` 讀，不落地）。
- 每次抓成功會存快取 `cache/ecount_master.json`（24h 內重跑直接用快取，
  避免頻繁登入觸發 ECOUNT 的 412 頻率限制）；API 失敗時自動回退上次快取。
- 極少數品項（如好市多特製版）主檔沒填 NO_USER1，該列「售價來源」標「推估」
  （供貨價 ÷ (1−15%)），儀表板上以 `*` 標記。

## 指令表怎麼讀

- **淨收入** ＝ 供貨價 − 定價 × 平台後毛利(10%)。真實毛利率＝(淨收入−成本)/淨收入。
  因為用逐品項的官方定價，平台前毛利是實算的（不再假設固定 15%）。
- **建議售價** 是實際操作的數字（取整到 10 元）；建議供貨價依該品項實際折數同步移動。
- What-if 試算滑桿同時顯示新建議售價、新供貨價、預期銷量、毛利額變化。

### 五個方向的意義

- **漲價／降價**：往利潤最大價（毛利率＝1/|彈性|）移動一步
  （單月上限 7%；用預設彈性的品牌減半為 3.5%）
- **贈品出清**：庫存過深（>12 個月）但毛利已低於目標 → 降價的打平公式分母太薄、
  換不到量，改用贈品／組合製造便宜感，不動售價
- **降價出清**：庫存過深但毛利仍厚 → 直接降價換量划算
- **檢視-負毛利**（紅底）：扣完平台 25% 後賣一台虧一台，優先人工處理。
  約 6 成是月銷 <3 的 EOL 清倉品（噪音），**月銷 ≥10 的才是真警訊**
- **不動**：已在最佳價附近，或已達競品天花板

**彈性來源**：品類（該品牌該品類自估，最準）→ 品牌 → 預設（-2.5 假設值，步幅減半）。
調價後隔月自動「急煞檢查」：實際銷量比預測差 50 個百分點以上者警示退回原價。

## 參數調整（config.json）

| 參數 | 目前值 | 說明 |
|---|---|---|
| platform_front/back_margin | 15% / 10% | 平台前後毛利，抽成改了就改這裡 |
| max_step | 7% | 單月最大調幅 |
| min_margin | 10% | 地板毛利，降價不會低於此線 |
| default_elasticity | -2.5 | 品牌估不出彈性時的假設值 |
| use_stock_logic | **true** | 庫存出清邏輯（已啟用：全通路銷量＋BQ 每日庫存讓庫存月數可信） |
| overstock_months | **12** | 滯銷門檻。igogo 備貨深（中位 8.4 個月），設 12 才抓得出真正該處理的 |
| comp_ceiling_factor | 1.05 | 漲價不得超過同形態競品中位 × 此倍數 |
| pricing_customers | MOMO, PChome | 定價只用這兩通路（抽成假設僅適用它們） |

## 已知限制

1. **BQ 庫存不一定每天更新**：`sync_inventory.py` 的每日排程若未建，`inventory_snapshot`
   會停在舊快照。庫存相關建議（贈品出清、庫存月數）要先確認快照日期；
   漲/降/負毛利不受影響。
2. **競品目前只有靜態錨定**：能當漲價天花板，但還算不出「權重 w」
   （消費者在意絕對價格 vs 你跟競品的價差，比重多少）。需競品快照累積 3~6 個月後，
   把 v_sales 與 competitor_prices 對齊跑 `ln(Q)=a+β_自·ln(P_自)+β_競·ln(P_競)`，
   w = |β_競|/|β_自|。爬蟲排程已建，時間到就能算。
3. **建議是方向與量級，不是精確預測**：彈性在現價附近估計，外推越遠越不可靠。
   實務上先調一步、觀察兩週，靠急煞機制校正。
4. **異常值過濾很關鍵**：剔除月均價 < 型號中位價 60% 的列（沖銷／特殊結價）。
   漏掉這步 JLab 彈性會從 −2.7 變 −1.1，結論完全相反。
